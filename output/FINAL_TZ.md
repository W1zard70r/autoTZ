# Техническое задание (IT-проект)

## 1. Общее описание проекта
- **Назначение системы:** Система предназначена для обеспечения безопасного доступа к ресурсам через VPN‑бот в Telegram, автоматической обработки и анализа документов с помощью LLM‑pipeline, а также для предоставления удобного веб‑интерфейса для управления и мониторинга процессов. Она ориентирована на команды разработчиков и аналитиков, которым необходимо быстро и надёжно обрабатывать большие объёмы данных, обеспечивая при этом защищённое соединение и масштабируемую архитектуру. Проблемы, которые решает система, включают сложность настройки VPN‑соединений, необходимость интеграции различных сервисов (S3, PostgreSQL, Redis) и потребность в гибкой, модульной обработке данных.
- **Аналоги/конкуренты:** ShukaVPN bot
- **Итоговый продукт:** В результате проекта заказчик получит полностью функционирующую инфраструктуру, включающую Telegram‑бота ShukaVPN для безопасного подключения, серверную часть на FastAPI, клиентский интерфейс на React, распределённую очередь задач Celery, ORM‑слой Prisma, PostgreSQL‑базу и Redis‑кеш, а также контейнеризированную сборку на Docker. Кроме того, будет реализован модульный LLM‑pipeline, парсер документов, потоковое чтение данных из Amazon S3 с помощью aiobotocore и ffmpeg, а также набор диаграмм архитектуры, созданных в Mermaid. Это позволит заказчику быстро развернуть, масштабировать и поддерживать сервис, обеспечивая безопасность, производительность и удобство использования.

## 5. Нефункциональные требования
### Безопасность
- Все клиентские соединения с сервисом будут защищены протоколом HTTPS, что обеспечивает шифрование данных при передаче и защищает от MITM-атак.
- Для аутентификации и авторизации пользователей будет использоваться JWT (JSON Web Tokens). Токены выдаются после успешной авторизации через OAuth‑2.0 и подписываются секретным ключом, хранящимся в защищённом хранилище (HashiCorp Vault / AWS Secrets Manager).
- Внутренние сервисы (FastAPI, Celery workers, Redis, PostgreSQL) будут доступны только через VPN‑канал, подключённый к ShukaVPN bot. Это гарантирует, что доступ к данным возможен только из доверенной сети.
- Все чувствительные данные (пароли, ключи доступа к S3, токены) хранятся в зашифрованном виде в базе PostgreSQL и Redis, а также в переменных окружения, управляемых Docker‑secret.
- Для защиты от CSRF и XSS применяются стандартные механизмы FastAPI (CORS‑политика, CSRF‑токены) и React‑приложения (Content Security Policy).
- Регулярные аудиты кода и статический анализ (Bandit, SonarQube) проводятся в CI/CD pipeline, чтобы выявлять уязвимости до релиза.
### Масштабируемость
- Архитектура разделена на независимые модули (парсер, обработка аудио, LLM‑инференс), каждый из которых упакован в Docker‑контейнер и может масштабироваться горизонтально через Docker‑Swarm / Kubernetes.
- Celery workers используют Redis как брокер сообщений, что позволяет динамически увеличивать количество воркеров в зависимости от нагрузки (например, при росте количества задач по обработке аудио).
- FastAPI работает в асинхронном режиме (uvicorn + async/await), поэтому один экземпляр может обслуживать сотни одновременных запросов, а при необходимости можно запустить несколько экземпляров за балансировщиком нагрузки.
- S3‑streaming с aiobotocore обеспечивает потоковую передачу больших файлов без загрузки их полностью в память, что снижает нагрузку на сервер и позволяет обрабатывать несколько потоков одновременно.
- Ожидаемая нагрузка: до 200 активных пользователей одновременно, 10 000 запросов в день на API, 5 000 задач в очереди Celery в сутки. При такой нагрузке система должна сохранять стабильность, а горизонтальное масштабирование позволит увеличить лимиты до 1 000 пользователей и 100 000 запросов в сутки без изменения кода.
### Время отклика
- Для API‑эндпоинтов FastAPI установлена целевая SLA: 95 % запросов должны отвечать за 200 мс или меньше. Это достигается благодаря асинхронной обработке, кешированию в Redis и оптимизации запросов к PostgreSQL.
- Время выполнения фоновых задач Celery (парсинг, извлечение аудио, инференс LLM) ограничено 5 секундами в 95 % случаев. При превышении лимита задача помечается как «failed» и автоматически перезапускается.
- Время отклика пользовательского интерфейса React не должно превышать 300 мс для основных действий (загрузка списка задач, просмотр статуса). Для долгих операций используется WebSocket‑обновление статуса.
- Тесты производительности (Locust, k6) проводятся в CI/CD, чтобы гарантировать, что новые релизы не ухудшают показатели отклика.
