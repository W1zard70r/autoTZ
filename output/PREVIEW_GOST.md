## 2. Назначение и цели

# Назначение и цели создания системы

Система предназначена для автоматизации корпоративного обучения сотрудников с возможностью назначения курсов администратором через личный кабинет. Основная цель — создание централизованной платформы, где администратор может управлять учебным контентом, просматривать аналитику по успеваемости сотрудников и контролировать процесс обучения в реальном времени.

Система реализует полный цикл корпоративного обучения: сотрудники могут приобретать подписку через интегрированный магазин курсов с поддержкой рекуррентных платежей через Stripe, отслеживать свой прогресс через персональный дашборд и получать доступ к назначенным курсам. Для технической реализации предусмотрена разработка REST API с эндпоинтами POST /api/v1/courses/assign для назначения курсов и GET /api/v1/users/me/progress для получения прогресса обучения, а также выбор оптимальной технологии бэкенда и СУБД для обеспечения стабильной работы MVP-версии продукта.

Система должна соответствовать следующим требованиям:

### 3.1. Функциональные требования

# Функциональные требования

## Аутентификация и авторизация
Система должна обеспечивать аутентификацию пользователей с использованием JWT токенов. Access токен должен иметь время жизни 15 минут, refresh токен - 7 дней. При истечении access токена пользователь должен автоматически получать новый токен через refresh токен.

## Административный интерфейс
Личный кабинет для Администратора должен предоставлять возможность управления пользователями, курсами и настройками системы. Администратор должен иметь доступ ко всем функциям управления контентом и настройками монетизации.

## Панель управления
Дашборд с прогрессом должен отображать статистику по курсам, активным подпискам, доходам и ключевым метрикам системы в реальном времени. Дашборд должен обновляться автоматически каждые 30 секунд.

## Магазин курсов
Магазин курсов должен предоставлять каталог с фильтрацией по категориям, ценам и рейтингу. Каждый курс должен содержать описание, программу, отзывы и возможность покупки.

## Монетизация
Система должна поддерживать рекуррентную подписку с возможностью выбора периодичности (месячная, годовая). Подписка должна автоматически продлеваться по окончании срока действия.

## Платежная система
Интеграция Stripe должна обеспечивать прием платежей по банковским картам, Apple Pay и Google Pay. Система должна поддерживать возвраты и возврат средств через Stripe API.

### 3.2. Стек технологий

# Технологический стек

## Серверная часть

В качестве технологии серверной части системы используется **Python + Django**. Данная технология обеспечивает надежную и масштабируемую основу для разработки веб-приложений. В качестве альтернативного решения рассматривается **Node.js + NestJS**, который также может быть использован для реализации серверной логики.

## База данных

Для хранения структурированных данных предусмотрена **PostgreSQL** - реляционная система управления базами данных, обеспечивающая ACID-совместимость и поддержку сложных запросов. Для хранения документов и неструктурированных данных используется **MongoDB** - документоориентированная СУБД, обеспечивающая гибкость схемы данных.

## Интерфейсы

Взаимодействие между клиентом и сервером осуществляется через **REST API** - архитектурный стиль, основанный на представлении ресурсов и использовании стандартных HTTP-методов. Для обеспечения безопасности передачи данных используется **httpOnly cookie**, который хранится на стороне клиента и недоступен для JavaScript-кода, что защищает от XSS-атак.
